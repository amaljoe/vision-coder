\begin{abstract}
We present \model{VisionCoder}, a system for generating HTML/CSS code from webpage screenshots by fine-tuning a 2-billion-parameter vision-language model with reinforcement learning from verifiable visual rewards. The core technical challenge is that standard language modelling objectives do not optimise for \emph{rendered visual appearance} — a model can produce syntactically valid but visually empty HTML and incur no training loss. We address this by rendering each generated HTML page in a headless browser and comparing the screenshot to the reference image via CLIP visual similarity, using this as the primary reward signal (weighted 3$\times$) alongside three auxiliary rewards for code format, HTML validity, and structural DOM similarity. All rewards are computed automatically without human annotation. We further adopt a resolution curriculum during supervised fine-tuning (SFT): training 60\% of steps at quarter resolution ($\approx$12 image tokens), 20\% at half resolution ($\approx$48 tokens), and 20\% at full resolution ($\approx$192 tokens), before transitioning to Group Relative Policy Optimisation (GRPO) fine-tuning at full resolution. On the Design2Code benchmark (484 real-world webpages, \citealp{si2025design2code}), \model{VisionCoder-SFT+RL} achieves an overall score of \textbf{0.802}, matching Gemini Pro Vision (0.801) and surpassing GPT-4V (0.796) while using $\mathbf{>99\%}$ fewer parameters. Critically, we show that SFT warm-starting is essential: RL trained directly from the base model reaches only 0.354 because the policy collapses into degenerate attractors. Detailed analysis of training dynamics reveals a characteristic generation-length lifecycle, a visual awakening phase transition at step $\approx$100, and differential entropy responses to visual versus structural reward signals.
\end{abstract}

\section{Introduction}
%% ============================================================

Modern software development is increasingly visual: designers produce high-fidelity mockups and screenshots, and front-end engineers must translate these into HTML, CSS, and JavaScript. This translation is time-consuming, error-prone, and requires detailed knowledge of both visual design and web standards. Automating even part of this workflow — producing a first-pass HTML implementation from a screenshot — would substantially reduce the cost of front-end development.

Vision-language models (VLMs) have demonstrated impressive capabilities on many visual understanding tasks, including image captioning, visual question answering, and document understanding~\citep{liu2023llava,dai2023instructblip,alayrac2022flamingo}. Screenshot-to-code generation has emerged as a natural test of these capabilities: it requires the model to simultaneously parse the visual layout, identify UI elements and their spatial relationships, and generate syntactically correct, semantically meaningful markup~\citep{si2025design2code,huggingface2024websight}.

However, standard fine-tuning approaches face a fundamental limitation: the cross-entropy language modelling objective optimises token-level prediction accuracy, not the visual fidelity of the rendered output. A model can learn to generate plausible-looking HTML — correctly formatted, with proper tags — yet produce a page that renders as an empty white rectangle. There is no gradient signal pushing the policy toward visual correctness unless the model is explicitly rewarded for what the generated code \emph{looks like} when executed.

We address this gap by casting screenshot-to-HTML generation as a reinforcement learning problem with verifiable visual rewards. Rather than comparing predicted and reference tokens, we \emph{render} the predicted HTML using a headless Playwright browser~\citep{playwright2023} and compare the resulting screenshot to the target using CLIP image embeddings~\citep{radford2021clip}. This closing of the render loop provides a direct, differentiable signal toward visual fidelity. We call our system \model{VisionCoder}.

The core training pipeline follows the pattern established by DeepSeek-R1~\citep{deepseek2025r1} for reasoning tasks: begin with supervised fine-tuning (SFT) on a curated dataset to establish a strong initial policy, then apply GRPO~\citep{shao2024deepseekmath} to refine the policy toward a multi-component verifiable reward. Critically, we show that this two-stage approach is essential for visual generation: RL from the raw base model reaches only 0.354 on Design2Code, while SFT+RL achieves 0.802 — a $2.3\times$ improvement attributable entirely to the SFT warm-start.

\paragraph{Contributions.} We make the following contributions:

\begin{itemize}[leftmargin=1.5em, itemsep=2pt]
  \item \textbf{A multi-component rendering-based reward} combining code format, HTML validity, DOM structural similarity, and CLIP visual similarity (3$\times$ weight), all computed automatically without human labels. The theoretical reward ceiling is 6.0; trained models reach $\approx5.1$ in-distribution.
  \item \textbf{A resolution curriculum for SFT} that trains the model progressively from coarse (128\,px, $\approx$12 image tokens) to full resolution (512\,px, $\approx$192 image tokens), reducing early-stage compute by up to 16$\times$ while letting the model first learn HTML structure before visual fidelity.
  \item \textbf{Competitive results at 2B scale:} \model{VisionCoder-SFT+RL} achieves an overall score of 0.802 on Design2Code, matching Gemini Pro Vision and surpassing GPT-4V, and outperforming GPT-4V by 41\% on block-level layout matching.
  \item \textbf{Detailed training dynamics analysis} identifying a four-phase generation-length lifecycle, a visual awakening phase transition, and the mechanism by which SFT warm-starting prevents policy collapse.
\end{itemize}

\section{Related Work}
%% ============================================================

\subsection{Screenshot-to-Code Generation}

The task of generating UI code from visual inputs has a long history. \citet{beltramelli2017pix2code} introduced the problem with a CNN–RNN model trained on synthetic GUI screenshots, achieving 77\% accuracy on simple iOS, Android, and web layouts. While pioneering, this approach was limited to constrained domain-specific languages and toy layouts.

\citet{huggingface2024websight} scaled the problem to real HTML/CSS by releasing WebSight, a dataset of 2 million rendered screenshots paired with their generating HTML. They trained Sightseer, a VLM fine-tuned on this data, demonstrating that large-scale synthetic data can teach models to generate real web markup. Our work builds directly on this dataset.

The Design2Code benchmark~\citep{si2025design2code} provides 484 real-world webpages from Common Crawl and establishes a rigorous, rendering-based evaluation protocol that compares predicted and reference pages at the level of detected text blocks, their positions, colours, and overall CLIP similarity. This benchmark, and its OCR-free evaluation pipeline, forms the primary evaluation in our work. The original paper found that GPT-4V was the top-performing model at the time of publication, with an overall score of 0.796, and that a fine-tuned 18B model (Design2Code-18B) based on CogAgent could match Gemini Pro Vision in overall performance.

\subsection{Reinforcement Learning from Verifiable Rewards}

The use of reinforcement learning to fine-tune language models has a rich history beginning with RLHF~\citep{ouyang2022instructgpt}, which used human preference annotations and PPO~\citep{schulman2017proximal} to align GPT-3 with human intent. The InstructGPT work demonstrated that even a 1.3B model aligned with RLHF could outperform an unaligned 175B model on human preference tasks.

Subsequent work has moved toward \emph{verifiable} rewards — signals that can be computed automatically from ground-truth answers without human annotation. DeepSeekMath~\citep{shao2024deepseekmath} introduced Group Relative Policy Optimisation (GRPO), an efficient alternative to PPO that eliminates the critic network by using within-group reward statistics to estimate advantages. GRPO has since become widely adopted for its simplicity and efficiency.

DeepSeek-R1~\citep{deepseek2025r1} showed that GRPO with verifiable rewards — using rule-based correctness checks for mathematics — can dramatically improve reasoning capabilities, achieving state-of-the-art results on AIME with a 71\% solve rate. Crucially, the DeepSeek-R1 work demonstrated both a pure-RL approach (R1-Zero, trained from scratch with RL) and a warm-start approach (R1, with SFT cold-start before RL). The warm-start approach consistently outperformed pure RL, a finding that our work corroborates in the visual generation domain.

DPO~\citep{rafailov2023dpo} offers an alternative to RL-based alignment by reframing preference learning as a classification objective, avoiding explicit reward models. While simpler to train, DPO requires paired preference data and does not naturally accommodate the rendering-based rewards central to our work.

\subsection{Vision-Language Models}

Large-scale VLMs have been the primary tool for multimodal understanding tasks. LLaVA~\citep{liu2023llava} demonstrated that connecting a pre-trained visual encoder to a large language model and fine-tuning on instruction-following data yields strong multimodal performance. InstructBLIP~\citep{dai2023instructblip} introduced query-adaptive visual feature extraction, enabling more flexible visual reasoning. Flamingo~\citep{alayrac2022flamingo} pioneered in-context few-shot learning for VLMs with interleaved visual-text sequences.

Qwen-VL~\citep{bai2023qwenvl} introduced a competent open-weight VLM with strong spatial understanding and text-reading capabilities, particularly relevant for web screenshots that contain structured text layouts. Qwen3-VL~\citep{qwen3vl2025} — the base model in our work — extends this with a DeepStack visual architecture and interleaved Multi-scale RoPE positional encodings, supporting dynamic resolution processing and enabling efficient tokenisation at multiple image resolutions.

\subsection{Visual Similarity as a Reward Signal}

CLIP~\citep{radford2021clip} demonstrated that contrastive pre-training on 400 million image-text pairs yields visual representations that capture semantic and perceptual similarity. While originally designed for image-text alignment, CLIP embeddings have proven effective for image-image similarity, forming the basis of the visual reward in our system.

\citet{zhang2018lpips} showed that deep neural network features outperform pixel-level metrics such as SSIM for perceptual similarity evaluation. This insight motivates our use of high-level CLIP features rather than pixel comparison for reward computation. Using semantically meaningful representations ensures that the reward signal captures visual fidelity at the level of layout and content, not pixel-perfect reproduction.

\section{Methodology}
%% ============================================================

\subsection{Task Formulation}

Given a screenshot $\mathbf{I} \in \mathbb{R}^{H \times W \times 3}$ of a webpage, the task is to generate an HTML/CSS string $h$ such that rendering $h$ in a browser produces a page that visually matches $\mathbf{I}$. Formally, we model a policy $\pi_\theta$ that maps a prompt $x = [\text{sys}, \mathbf{I}, \text{query}]$ to a completion $h \sim \pi_\theta(\cdot \mid x)$.

The system prompt instructs the model to act as an expert web developer and to output the complete HTML with inline CSS, using \texttt{rick.jpg} as a placeholder for any images. The user query is: \emph{``Generate the HTML code that reproduces this website screenshot.''}

Following Qwen3-VL's instruction format, the model is prompted with thinking mode disabled, as chain-of-thought tokens would consume generation budget without contributing to the HTML output.

\subsection{Base Model}

We build on \model{Qwen3-VL-2B-Instruct}~\citep{qwen3vl2025}, a 2-billion-parameter vision-language model with a DeepStack visual architecture. Images are tokenised with patch size 16 and spatial merge factor 2 (effective patch resolution: $32 \times 32$ pixels per token). At our training resolution of 512\,px maximum width, a typical webpage screenshot occupying roughly $512 \times 384$ pixels requires approximately 192 image tokens — a compact but semantically sufficient representation for webpage layout understanding.

The model uses interleaved Multi-scale Rotary Position Embeddings (MRoPE) to encode the two-dimensional spatial structure of image patches alongside one-dimensional text positions, enabling the decoder to reason about the spatial arrangement of visual elements.

\subsection{Multi-Component Reward Design}
\label{sec:reward}

The total reward at each GRPO step is a weighted sum of four components:
\begin{equation}
  R = R_{\mathrm{fmt}} + R_{\mathrm{val}} + R_{\mathrm{struct}} + 3 \cdot R_{\mathrm{clip}},
  \label{eq:reward}
\end{equation}
giving a theoretical maximum of $1 + 1 + 1 + 3 = 6.0$. The 3$\times$ weighting on $R_{\mathrm{clip}}$ makes visual fidelity the dominant signal (50\% of maximum reward weight), with the remaining components serving as structural floor constraints that encourage syntactic validity before semantic quality.

\paragraph{Format Reward ($R_{\mathrm{fmt}}$, max 1.0).}
This binary-style reward checks whether the output conforms to the expected response structure:
\begin{align*}
  R_{\mathrm{fmt}} &= 0.5 \cdot \mathbf{1}[\text{\texttt{```html} fence present}] \\
                   &\quad + 0.25 \cdot \mathbf{1}[\text{\texttt{<html}/\texttt{<!DOCTYPE} tag present}] \\
                   &\quad + 0.25 \cdot \mathbf{1}[\text{\texttt{</html>} closing tag present}].
\end{align*}
This reward converges to near-perfect ($\geq0.99$) within the first 50 GRPO steps, indicating that the model quickly learns the structural contract.

\paragraph{Validity Reward ($R_{\mathrm{val}}$, max 1.0).}
HTML is parsed with the \texttt{html5lib} tolerant parser. Three sub-scores assess:
\begin{align*}
  R_{\mathrm{val}} &= 0.4 \cdot \mathbf{1}[\text{parse succeeds}] \\
                   &\quad + 0.3 \cdot \tfrac{1}{3}\bigl(\mathbf{1}[\texttt{<html>}] + \mathbf{1}[\texttt{<head>}] + \mathbf{1}[\texttt{<body>}]\bigr) \\
                   &\quad + 0.3 \cdot \min\!\bigl(\tfrac{|\mathcal{T} \cap \mathcal{K}|}{6}, 1\bigr),
\end{align*}
where $\mathcal{T}$ is the set of distinct tags in the output and $\mathcal{K}$ is a set of 16 key structural tags (\texttt{div}, \texttt{p}, \texttt{span}, \texttt{a}, \texttt{table}, etc.). The diversity component prevents degenerate outputs that technically parse but contain only a single tag type.

\paragraph{Structural Reward ($R_{\mathrm{struct}}$, max 1.0).}
DOM structural similarity is measured by comparing the tag-level sequence of the prediction against the reference HTML:
\begin{equation}
  R_{\mathrm{struct}} = \frac{2 \cdot \mathrm{LCS}(S_{\mathrm{pred}}, S_{\mathrm{ref}})}{|S_{\mathrm{pred}}| + |S_{\mathrm{ref}}|},
\end{equation}
where $S$ denotes the BFS-ordered sequence of HTML tags (truncated to 200 tags to bound computation) and LCS is the longest common subsequence. CSS class overlap (Jaccard similarity) is also computed but used as a secondary signal.

\paragraph{Visual Reward ($R_{\mathrm{clip}}$, max $\approx$0.95).}
Each generated HTML string is rendered by a headless Chromium browser (via Playwright~\citep{playwright2023}) at 1280$\times$1024 pixels in an async pool of up to 8 concurrent instances, with a 5-second timeout per page. The rendered screenshot is compared to the reference image using CLIP (\texttt{openai/clip-vit-large-patch14}):
\begin{equation}
  R_{\mathrm{clip}} = \cos\!\bigl(\phi(\mathbf{I}_{\mathrm{pred}}),\; \phi(\mathbf{I}_{\mathrm{ref}})\bigr),
\end{equation}
where $\phi$ denotes the CLIP image encoder. The 3$\times$ boost in Equation~\eqref{eq:reward} reflects our finding that structural rewards alone are insufficient to drive visual quality: a model can achieve $R_{\mathrm{fmt}} = R_{\mathrm{val}} = 1.0$ and $R_{\mathrm{struct}} \approx 0.5$ by generating a structurally plausible but visually empty white-page skeleton, without ever touching $R_{\mathrm{clip}}$.

\subsection{SFT Pre-Training with Resolution Curriculum}
\label{sec:sft}

Before GRPO, we pre-train the model with supervised fine-tuning on WebSight~\citep{huggingface2024websight} to establish a strong initial policy. Training from the raw base model with GRPO directly leads to policy collapse (Section~\ref{sec:coldstart}), as the model's initial HTML generation policy is too diffuse to benefit from reward gradients.

\paragraph{Resolution Curriculum.}
Full-resolution images require $\approx$192 image tokens at 512\,px width, but early SFT steps need only teach HTML structure and syntax — tasks for which a rough sketch of the page layout suffices. We therefore train with progressively increasing resolution:

\begin{table}[H]
  \centering
  \caption{Resolution curriculum used during SFT. Image token counts are for a typical 4:3 aspect ratio screenshot. Training 60\% of steps at quarter resolution reduces image token cost by up to 16$\times$ for the majority of training.}
  \label{tab:curriculum}
  \small
  \begin{tabular}{lcccp{4cm}}
    \toprule
    \textbf{Phase} & \textbf{Steps} & \textbf{Width} & \textbf{$\approx$ Image Tokens} & \textbf{Training Focus} \\
    \midrule
    1 (60\%) & 0--1{,}500   & 128\,px & $\approx$12  & HTML syntax and document structure \\
    2 (20\%) & 1{,}500--2{,}000 & 256\,px & $\approx$48  & Element layout and text content \\
    3 (20\%) & 2{,}000--2{,}500 & 512\,px & $\approx$192 & Visual fidelity and colour matching \\
    \bottomrule
  \end{tabular}
\end{table}

The 16$\times$ reduction in image tokens during Phase 1 (12 vs.\ 192 tokens) dramatically reduces per-step GPU memory and computation, allowing efficient training on 20,000 WebSight samples. The curriculum ensures that the model has already learned robust HTML generation before the high-resolution visual detail needed for CLIP-level fidelity is introduced.

Image dimensions are aligned to the model's 32-pixel grid (patch size 16, merge factor 2) using Lanczos downsampling. At full resolution (512\,px wide), a typical 4:3 webpage renders to $512\times384$ pixels, occupying $16 \times 12 = 192$ image token positions in the model's input sequence.

\paragraph{SFT Training Details.}
We train for 2,500 steps with learning rate $2\times10^{-5}$, batch size 4, cosine schedule with 5\% warmup, and apply loss only to assistant completions (completion-only loss). Gradient checkpointing is used throughout.

\subsection{GRPO Fine-Tuning}

Following SFT, we fine-tune with GRPO~\citep{shao2024deepseekmath} for 1,000 steps. For each prompt, we sample $G=8$ completions from the current policy and compute group-relative advantages:
\begin{equation}
  \hat{A}_i = \frac{r_i - \mu_r}{\sigma_r}, \quad \mu_r = \frac{1}{G}\sum_{j=1}^{G} r_j, \quad \sigma_r = \sqrt{\frac{1}{G}\sum_{j=1}^G (r_j - \mu_r)^2},
\end{equation}
where $r_i = R(h_i, \mathbf{I})$ is the total reward for the $i$-th completion. This group-relative normalisation eliminates the need for a separate critic network, substantially reducing memory footprint. We use the DR-GRPO loss variant with $\beta=0$ (no KL penalty against a reference policy), which we found to improve training speed by $\approx$38\% compared to KL-regularised GRPO without degrading final performance.

Maximum completion length is set to 2,048 tokens. The learning rate is $1\times10^{-5}$ with batch size 8 (4 per device) and 2 gradient accumulation steps. GRPO generation is performed using vLLM~\citep{vonwerra2022trl} in collocated mode, sharing GPU memory with the training process on 2$\times$ A100 80\,GB GPUs.

\subsection{Implementation Details}

\paragraph{Hardware and Framework.}
Training uses 2$\times$ NVIDIA A100 80\,GB GPUs (NVLink) managed by Accelerate~\citep{gugger2022accelerate} with DeepSpeed ZeRO-2. Models are trained in bfloat16 with Flash Attention 2 where available, falling back to scaled-dot-product attention otherwise. The full pipeline uses TRL 0.28.0~\citep{vonwerra2022trl}, Transformers 4.57.6~\citep{wolf2020transformers}, and vLLM 0.15.0.

\paragraph{Rendering.}
The browser pool maintains up to 8 concurrent Playwright Chromium instances, each rendering at $1280\times1024$ viewport resolution with a 5-second timeout. Rendering is performed asynchronously so that multiple rollouts can be scored in parallel during GRPO data collection.

\paragraph{CLIP.}
We use \texttt{openai/clip-vit-large-patch14} as the visual similarity oracle. Both the generated screenshot and the reference are centre-cropped and normalised to $224\times224$ before CLIP encoding. Similarity is computed as cosine distance in the image embedding space.

\section{Experiments}
%% ============================================================

\subsection{Experimental Setup}

\paragraph{Training Data.}
SFT training uses 20,000 samples from WebSight v0.2~\citep{huggingface2024websight}, filtered to examples with fewer than 3,000 HTML characters and shuffled with seed 42. GRPO uses 2,000 samples from the same pool. All WebSight images are synthetically generated; the model never sees real-world webpage screenshots during training.

\paragraph{Evaluation.}
We evaluate on two benchmarks:
\begin{itemize}[leftmargin=1.5em, itemsep=2pt]
  \item \textbf{Design2Code}~\citep{si2025design2code}: 484 real-world webpages with ground-truth HTML. Evaluation renders predictions to PNG and measures five metrics: Block-Match (layout-level element matching), Text (character-level text fidelity), Position (spatial coordinate accuracy), Color (RGB colour similarity), and CLIP (rendered-image visual similarity). The Overall score is the arithmetic mean of these five components. We use the same evaluation script as Si et al.~\citeyearpar{si2025design2code}.
  \item \textbf{WebSight holdout}: 100 held-out samples from the end of the WebSight training distribution, evaluating Format, Validity, Structural, and CLIP rewards as defined in Section~\ref{sec:reward}. The Total score follows the weighting in Equation~\eqref{eq:reward}.
\end{itemize}

\paragraph{Baselines.}
We compare against three baselines:
\begin{itemize}[leftmargin=1.5em, itemsep=2pt]
  \item \textbf{Qwen3-VL-2B (base)}: Zero-shot generation with the Design2Code direct-prompting template from \citet{si2025design2code}, without any fine-tuning.
  \item \textbf{GPT-4V and Gemini Pro Vision}: Results from \citet{si2025design2code} using the same direct-prompting setup and evaluation protocol. Overall scores are computed as the arithmetic mean of the five Design2Code component metrics, consistent with our evaluation.
  \item \textbf{VisionCoder-RL}: Our ablation model, trained with GRPO from the base model \emph{without} SFT warm-starting, for 1,000 steps.
\end{itemize}

This result is particularly striking because VisionCoder was trained entirely on \emph{synthetic} WebSight data, with no exposure to real-world webpages from Common Crawl during training.

\subsection{Qualitative Analysis}

Figure~\ref{fig:qualitative} shows three examples from Design2Code (indices 158, 5, and 428) comparing rendered outputs from the base model, VisionCoder-RL, and VisionCoder-SFT+RL against the reference screenshot.

On examples 158 and 5, both the base model and VisionCoder-RL produce near-blank white pages (overall scores of 0.154 and 0.150 respectively), while VisionCoder-SFT+RL faithfully reproduces the layout, typography, and colour scheme (0.967 and 0.925). This confirms that RL alone is insufficient: without SFT initialisation, the model's policy fails to escape the degenerate white-page attractor.

Example 428 is notable as the case where VisionCoder-RL \emph{does} produce meaningful output (score 0.896 vs.\ base 0.147), demonstrating that the RL model's failure is inconsistent rather than universal. VisionCoder-SFT+RL improves marginally further (0.919), with better colour and layout precision. This variability in VisionCoder-RL's performance — excellent on some inputs, completely failing on others — motivates the SFT warm-start as a means of stabilising the policy across the full input distribution.

\section{Training Dynamics Analysis}
%% ============================================================

We now present a detailed analysis of training dynamics, drawing on reward curves and completion-length statistics from GRPO training runs. These observations provide insight into \emph{how} the policy learns to generate visual HTML and identify failure modes to avoid in future work.

\subsection{The Generation-Length Lifecycle}
\label{sec:length}

One of the most striking observations in our training runs is a characteristic four-phase lifecycle in generation length, illustrated in Figure~\ref{fig:completions}.

\begin{figure}[t]
  \centering
  \includegraphics[width=\textwidth]{plots/completions.png}
  \caption{Generation length statistics (mean, min, max) over GRPO training steps. The four phases of the generation-length lifecycle are visible: (1) verbose exploration with frequent truncation, (2) brevity collapse as the model avoids truncation penalties, (3) visual awakening as CLIP reward incentivises richer HTML, and (4) stable expressive equilibrium. The CLIP-enabled run maintains longer completions ($\approx$456 tokens) than the no-CLIP run ($\approx$339 tokens), because richer HTML earns higher visual reward.}
  \label{fig:completions}
\end{figure}

\paragraph{Phase 1 — Verbose Exploration (steps 1--10).}
At the start of GRPO training, the model draws on its language modelling priors and generates verbose HTML: mean completion lengths of 950--1,450 tokens, with 3--37\% of rollouts hitting the 2,048-token maximum and being truncated mid-tag. Truncated HTML renders as broken or empty pages, incurring near-zero CLIP reward and low validity scores. The model initially has no mechanism to know how much HTML is ``enough.''

\paragraph{Phase 2 — Brevity Collapse (steps 10--100).}
The policy rapidly learns that shorter completions avoid the truncation penalty. Completion length drops precipitously to 200--400 tokens within the first 100 steps. However, the model initially compensates by generating structurally minimal HTML — valid skeletons with correct tags but little content, rendering as white pages. CLIP reward during this phase is near zero for most rollouts. The format and validity rewards, however, converge to near-perfect ($\geq 0.99$) by step 50, establishing the syntactic contract that subsequent learning builds upon.

\paragraph{Phase 3 — Visual Awakening ($\approx$step 100).}
In GRPO runs that include the CLIP visual reward, a phase transition occurs around step 100: the total reward jumps from $\approx$2.7 to $\approx$4.3 within a narrow window. This is the moment when the policy ``discovers'' that adding actual content (text, styled containers, coloured backgrounds) causes the rendered page to resemble the reference screenshot, yielding substantial CLIP reward. Structurally, this transition coincides with the model beginning to populate HTML with realistic content rather than empty tag scaffolds.

This phase transition has no counterpart in GRPO runs \emph{without} the CLIP reward (the no-CLIP run plateaus at $\approx$2.5, corresponding to format + validity + partial structural reward). \textbf{Without the visual CLIP signal, the model has no gradient toward rendering actual content}, and the policy stably converges to the white-page attractor from which it cannot escape.

\paragraph{Phase 4 — Expressive Equilibrium (steps 200+).}
Once the visual reward is unlocked, the policy enters a stable regime where it balances completeness with conciseness. For the CLIP-enabled run, completion lengths plateau around 340--460 tokens — substantially longer than the brevity-collapse minimum ($\approx$200 tokens), reflecting the policy's learned understanding that more expressive HTML earns higher visual reward. The no-CLIP run stabilises at $\approx$339 tokens with no upward pressure.

This lifecycle — verbose $\to$ brief $\to$ expressive — is an emergent consequence of the truncation-avoidance and visual fidelity incentives interacting over training time.

\subsection{Reward Phase Transitions}
\label{sec:reward_curves}

Figure~\ref{fig:rewards} plots the evolution of each reward component over training. Format and validity rewards behave identically across all runs: rapid convergence to near-1.0 within 50 steps. This confirms that both structural contract learning (correct code fences, valid tags) and basic HTML validity are easily mastered, freeing later training to focus on visual quality.

\begin{figure}[t]
  \centering
  \includegraphics[width=\textwidth]{plots/rewards.png}
  \caption{Per-component reward curves during GRPO training. Format and validity rewards (dashed) converge to near-perfect within 50 steps. The CLIP visual reward (orange) only activates in runs where it is included as a training signal; without CLIP, the total reward plateaus at $\approx$2.5. The phase transition at step $\approx$100 corresponds to the visual awakening described in Section~\ref{sec:length}.}
  \label{fig:rewards}
\end{figure}

The structural similarity reward ($R_{\mathrm{struct}}$) improves more gradually, reflecting the slower acquisition of layout-level HTML patterns. It reaches $\approx$0.77 at its peak in CLIP-enabled runs.

The CLIP visual reward exhibits the sharpest dynamics: near zero for the first 100 steps, then a rapid rise corresponding to the visual awakening phase, followed by stabilisation around 0.8 (unboosted; contributing $\approx$2.4 to total reward after 3$\times$ amplification). The total reward for CLIP-enabled runs stabilises at $\approx$5.0 out of a maximum of 6.0.

The absence of CLIP reward in the no-CLIP ablation run creates a striking outcome: total reward plateaus at $\approx$2.5 ($= 1.0_{\mathrm{fmt}} + 1.0_{\mathrm{val}} + 0.5_{\mathrm{struct}}$) and never advances. The model learns to game the structural reward with minimal HTML, achieving high format and validity scores while rendering as empty pages. This demonstrates that structural rewards alone are insufficient proxies for visual quality, and that direct rendering-based feedback is essential.

\subsection{The Necessity of SFT Warm-Starting}
\label{sec:coldstart}

The performance gap between VisionCoder-RL (0.354) and VisionCoder-SFT+RL (0.802) on Design2Code is entirely attributable to the SFT initialisation. To understand why, consider the nature of the policy at the start of GRPO training for each variant:

\textbf{Base model (no SFT).} The raw Qwen3-VL-2B-Instruct model has some instruction-following capability and can generate code-like text, but its HTML generation policy is diffuse: completions vary widely in quality, structure, and length. When GRPO begins, the 8 rollouts per prompt exhibit high variance in reward. The advantage estimates $\hat{A}_i$ are noisy, and the policy update signal is weak relative to the entropy in the current distribution. Furthermore, blank-page outputs (which score 0 on CLIP but may still score well on format and validity) represent a local maximum that the policy tends to converge toward: they are syntactically valid, short, and avoid truncation penalties.

\textbf{SFT-initialised model.} After 2,500 steps of SFT, the model's HTML generation policy is concentrated in the valid-HTML region. All 8 rollouts per prompt produce plausible HTML; variance is lower. More importantly, the range of rollout quality is semantically meaningful — some rollouts better match the reference visual layout than others — giving GRPO informative advantage estimates to act on. The policy is already beyond the white-page attractor, allowing GRPO to focus on refining visual quality rather than discovering basic HTML structure from scratch.

The evidence from WebSight holdout evaluation corroborates this mechanism. VisionCoder-RL achieves near-perfect Format (1.000) and Validity (0.984), indicating that it \emph{does} eventually learn to produce syntactically valid HTML — but it lags significantly on Structural similarity (0.709 vs.\ 0.818 for SFT+RL), suggesting that even after 1,000 GRPO steps, it has not fully recovered the layout diversity that SFT provides. The Design2Code results amplify this gap under distributional shift: real-world webpages are far more diverse than synthetic WebSight pages, and the policy must generalise layout patterns it has never encountered during training.

\subsection{Entropy Dynamics and Policy Stability}

Figure~\ref{fig:dynamics} shows the evolution of policy entropy throughout GRPO training.

\begin{figure}[t]
  \centering
  \includegraphics[width=\textwidth]{plots/training_dynamics.png}
  \caption{Training dynamics: policy entropy and loss over GRPO training steps. Entropy decreases monotonically in both runs, indicating convergence to a confident policy. The CLIP-enabled run maintains higher entropy ($\approx$0.16 at step 500) than the no-CLIP run ($\approx$0.07), reflecting the visual reward's incentive for exploring diverse HTML approaches across different reference screenshots.}
  \label{fig:dynamics}
\end{figure}

In both runs, entropy decreases monotonically throughout training: from $\approx$0.34 at step 1 to $\approx$0.07 at step 500 for the no-CLIP run, and to $\approx$0.16 for the CLIP-enabled run. Monotonically decreasing entropy indicates a healthy convergence toward a confident policy without collapse — the model is not repeating a single stereotyped output, but has identified a high-quality generation strategy that it applies consistently.

The higher final entropy of the CLIP-enabled run ($0.16$ vs.\ $0.07$) is consistent with the generation-length finding: the visual reward requires different amounts and types of HTML content for different reference screenshots, naturally maintaining more diversity in the policy. A layout-intensive webpage requires different HTML patterns than a text-heavy article page, and the CLIP reward provides per-example signal that keeps the policy from over-specialising.

\section{Conclusion}
%% ============================================================

We have presented \model{VisionCoder}, a system for screenshot-to-HTML generation that bridges the gap between visual fidelity and code correctness through reinforcement learning with verifiable rendering-based rewards. Our central contribution is demonstrating that a 2-billion-parameter vision-language model, trained entirely on synthetic WebSight data with GRPO and a CLIP-based visual reward, can match Gemini Pro Vision and surpass GPT-4V on the Design2Code benchmark — a result achieved without any real-world webpage data during training.

Our analysis yields several concrete lessons for practitioners:

\textbf{Visual rewards are non-negotiable.} Structural rewards alone (format, validity, DOM similarity) are insufficient — they allow the policy to converge to a white-page attractor that scores well on all structural metrics while producing nothing visible. Closing the render loop with a CLIP-based visual signal is the key driver of visual quality improvement.

\textbf{SFT warm-starting prevents policy collapse.} RL from the raw base model reaches only 0.354 on Design2Code versus 0.802 for SFT+RL — a $2.3\times$ gap attributable entirely to initialisation. The SFT warm-start concentrates the policy in the valid-HTML region before GRPO begins, ensuring that reward gradients act on semantically meaningful variance rather than syntactic exploration.

\textbf{Generation length is an informative diagnostic.} The four-phase lifecycle (verbose $\to$ brief $\to$ expressive $\to$ stable) reliably indicates whether the visual reward has been discovered. Monitoring completion length provides an early signal of whether the model has escaped the brevity-collapse phase.

\textbf{Resolution curricula reduce SFT compute substantially.} Training 60\% of SFT steps at quarter resolution (12 vs.\ 192 image tokens, a 16$\times$ reduction) allows the model to learn HTML syntax efficiently before introducing the full visual detail needed for CLIP-level fidelity.

\paragraph{Limitations.} Our models use only 2B parameters and are trained on synthetic WebSight data; larger models and more diverse training data would likely yield further gains, particularly on Position and colour matching where proprietary models currently hold an advantage. CLIP similarity, while a strong proxy, does not capture all dimensions of visual quality — micro-interactions, typography details, and animation are invisible to a single-frame CLIP comparison. Future work could explore process rewards at render time (e.g., rewarding iterative visual debugging), larger-scale training with real-world webpage data, and human preference reward models that capture fidelity dimensions beyond CLIP.
